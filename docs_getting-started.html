1. Include the ZapCaptcha JS

This is the only required dependency. Place this near the end of your document <body> to access the library:

HTML (REQUIRED)
<script type="module" src="https://zapcaptcha.com/zapcaptcha.js"></script>


‚ÄºÔ∏èDeactivating javascript can bypass some gates. Put this guard in the <head> of your html file:

HTML (RECOMMENDED)
<noscript><meta http-equiv="refresh" content="0; url=/no-js.html"></noscript>

This optional line can go in your <head> section for best security, performance, and determinism:

HTML (RECOMMENDED)
<noscript><link rel="stylesheet" href="https://zapcaptcha.com/zapcaptcha.css"></noscript>
‚ö†Ô∏èYou can squeeze out a negligible increase in performance by preloading the styles, but that violates CSR unless allowing unsafe in-line. Hence, we don't recommend doing this:
<link rel="preload" href="https://zapcaptcha.com/zapcaptcha.css" as="style" onload="this.rel='stylesheet'">

It's rare but if you're having issues with a highly specialized mobile layout, put this line in your <head> section:

HTML (OPTIONAL)
<meta name="viewport" content="width=device-width, initial-scale=1.0, maximum-scale=1.0, user-scalable=no">

4. Set ZapCaptcha Options

Use metatag passed options in the HTML to customize your CAPTCHA.


Overview

Users can override default parameters using the zap-flags metatag and passing flags in HTML. Multiple flags can be set in a single metatag.


viewportConfined (Default: True)

By default, ZapCaptcha restricts the challenge hitbox spawn and exploration space to the viewport. This is to prevent frustration with having to find an offscreen hitbox.

However, this option is not set in stone and can be toggled using viewportConfined. To turn this safegaurd off, use the following metatag syntax:

<meta name="zap-flags" content="viewportConfined = false">

Turning off viewportConfined allows for a slightly more explorative and challenging CAPTCHA that sacrifices user experience to lower bot success rate. Of course, keep in mind unconfined viewport mode is meaningless if the document height is less than or equal to the viewport height.


canvasMode (Default: True)

By default, ZapCaptcha uses canvas drawing to provide its gamified CAPTCHA challenge. This provides an added layer of security and bot evasion. However, on some legacy systems, switching to DOM mode may be warranted. This can be done using the canvasMode flag. To activate legacy DOM CAPTCHA mode, use the following metatag syntax:

<meta name="zap-flags" content="canvasMode = false">

debugMessages (Default: False)

Debugging feature we left in. Enabling this option dumps potential console tamper signatures to the console. These checks contain high false positives and low accuracy, but together can be used by your team to possibly craft preventative measures.

<meta name="zap-flags" content="debugMessages = true">

lockoutsEnabled (Default: True)

This flag determines whether or not the ZapCaptcha client locks out users who perform flagged operations. Turn this off if you want to simply observe bot or threat actor activity and not immediately penailize it.

<meta name="zap-flags" content="lockoutsEnables = false">

softLock (Default: False)

This flag further refines lockout to not be a hard ban (the default) but rather a deactivation of CAPTCHAs and hence their gated buttons. For when you want locked out users to still be able to read and navigate.

<meta name="zap-flags" content="softLock = true">

sessionLock (Default: True)

This flag means lockout state gets stored in session storage. Can be combined with the cookie and local storage lockout flags. A light ban persistence.

<meta name="zap-flags" content="sessionLock = false">

cookieLock (Default: True)

This flag means lockout state gets stored in cookie storage. Can be combined with the session and local storage lockout flags. A light ban persistence.

<meta name="zap-flags" content="cookieLock = false">

localLock (Default: False)

This flag means lockout state gets stored in local storage. Can be combined with the session and local storage lockout flags. A heavy ban persistence.

<meta name="zap-flags" content="localLock = true">

serverMode (Default: False)

This flag enables the high-assurance server mode. To read about how server mode works and how to leverage it, read here.

<meta name="zap-flags" content="serverMode = true">

serverLock (Default: False)

This flag enables the high-assurance server trust-anchored server to solely determine lockouts Requires serverMode and if serverLock is false but serverMode is true, the server lockout input gets OR'd with local determinations.

<meta name="zap-flags" content="serverMode = true">

clickDelay (Default: False)

By default CAPTCHA clicks immediately register with the system. The clickDelay feature allows for increased bot evasion by adding a randomized click delay. This countermeasure may mean more clicks are needed and sacrifice user experience.

<meta name="zap-flags" content="serverMode = true">

spawnDelay (Default: False)

By default the CAPTCHA box immediately spawns in the browser. The spawnDelay feature allows for increased bot evasion by adding a randomized spawn delay. This countermeasure may mean a slight wait is needed and sacrifice user experience.

<meta name="zap-flags" content="serverMode = true">

frictionLess (Default: False)

To leverage the frictionless capability built-into ZapCaptcha, you must enable it first as it's disabled by default.

<meta name="zap-flags" content="frictionLess = true">


Security configuration

By default, ZapCaptcha employs a minimum subset of security features that are not configurable. From there it is up to the developer to set the mitigations that they would like in place.

And setting all of the security flags at once is easy. To turn all countermeasures on, use the allsec flag like so: 

<meta name="zap-flags" content="allsec = true">

‚ÑπÔ∏è Setting the allsec flag to does NOT turn off the viewportConfined capability or other non-security items listed above.

Overriding allsec flag

The All Security Features On flag can be overridden by simply adding the specific flag to override in succession like so:

<meta name="zap-flags" content="allsec = true, lockoutsEnabled = false, vpnCheck = false>

In some cases when turning on all security flags is too restrictive and results in unintended resultant user experience consequences, another approach is to explicitly enable only the desired protections.

For example, to restrict user clicking to only interactive elements (buttons, forms, links, etc) while also enabling canvas spoofing protection, use the following metatag and flags:

<meta name="zap-flags" content="clickBlockEnforcement = true, canvasSpoofingCheck = true">

Just these two settings alone are a pretty resonable policy for most public-facing websites. But please do tailor ZapCaptcha security to your needs.


‚ÑπÔ∏è Setting the clickBlockEnforcement flag enforces policy but does not detect and lockout.


‚ÑπÔ∏è All other security flags result in user lockout when the threat criteria is met.

Integrity Checking

When using the checksumJS and checksumCSS flags, no enforcement will be done unless hashes are provided via metatags:

<meta name="zap-js-integrity" content="sha384-JS_HASH_HERE">
<meta name="zap-css-integrity" content="sha384-CSS_HASH_HERE">

These hashes, which need to be updated to reflect the latest hashes, establish a trust anchor. Once the metatags are provided and configuration flags are set, the ZapCaptcha inner workings will check the sha384 sum provided against an onload check. Any tampering to the source code will result in immediate feedback and, if enabled, lockout.
